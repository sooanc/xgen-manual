# 역할/권한 관리

본 챕터는 솔루션의 **권한 모델 전체 구조** 와, 조직 업무 단위에 맞춘 **역할(Role)** 정의·할당 절차를 다룹니다.

## 권한 모델 — 3 레이어로 분리되어 있다 { #permission-model }

XGEN 의 권한 체계는 *등급 / 역할 / 권한* 세 레이어가 **각각 독립적으로 동작**합니다. 흔히 한 단어로 묶여 쓰이지만 실제로는 의미·범위·관리 방법이 다릅니다. 아래 매트릭스 한 장으로 정리합니다.

| # | 레이어 | 답하는 질문 | 값의 형태 | 정의 주체 | 예시 | 어디서 효과 |
|---|---|---|---|---|---|---|
| **1** | **권한 등급 (Tier)** | 사용자가 좌상단 **관리 설정** 모드에 들어갈 수 있는가? | 단일 boolean 플래그 (`is_superuser`) | 시스템 (고정 2단계) | Standard User / SuperUser | 좌상단 **관리 설정** 모드 전환 버튼 가시성 |
| **2** | **역할 (Role)** | 같은 SuperUser 라도 어떤 관리 메뉴를 보는가? | 다중 라벨 배열 | 조직이 자유롭게 정의 | 시스템 관리자, 거버넌스 담당자, 분석가, 운영자, 컴플라이언스 담당, … | SuperUser 사이드바의 노출 메뉴 범위 |
| **3** | **권한 (Permission)** | 각 화면·섹션·탭·버튼을 사용할 수 있는가? | ABAC 키 배열 | 권한 카탈로그(고정) → 역할 또는 개별 사용자에 매핑 | `admin.user:read`, `main.agentflow:manage`, `admin.governance-pii:*` | 화면 안의 섹션·탭·버튼 단위 게이팅 |

**레이어 간 의존성**

- **2 → 1**: *역할* 은 SuperUser 등급 위에서만 의미를 가집니다. Standard User 에게 *시스템 관리자* 역할을 부여해도 관리 설정 모드 자체에 들어갈 수 없으므로 효과가 없습니다.
- **3 → 1**: *권한* 은 Standard User 에게도 부여 가능합니다. 예를 들어 *Agent 개발자* 는 Standard User 등급 위에 `main.agentflow:*` / `main.tool:*` / `main.knowledge:*` 권한이 부여된 사용자입니다.
- **3 → 2**: *권한* 은 역할에 묶이거나 사용자에게 직접 부여됩니다. 보통 역할 단위로 묶고, 예외 케이스만 개별 권한으로 보강하는 것을 권장합니다.

### 자주 헷갈리는 경계 정리

- **"관리자(Administrator)" 라는 중간 등급은 존재하지 않습니다.** 좌상단 *관리 설정* 모드에 들어갈 수 있는 사람은 모두 `is_superuser: true` 인 **SuperUser** 한 등급이며, 그 안에서 *시스템 관리자* / *거버넌스 담당자* 같은 **역할** 로 책임이 갈립니다. 등급 전체 정의는 [권한 등급 — 2단계 모델](20-admin-overview.md#permission-tiers), 두 역할의 메뉴 범위 차이는 [관리자 콘솔 개요 도입부](20-admin-overview.md) 의 *SuperUser 권한 필요* 안내를 참고하세요.
- **"초기(루트) SuperUser" 도 별도 등급이 아닙니다.** 솔루션 최초 설치 후 SuperUser 가 한 명도 없을 때 부트스트랩 경로(`/admin/create-superuser`) 로 만든 첫 SuperUser 일 뿐이며, 이후 사용자 관리 모달에서 *유형 = Superuser* 로 추가되는 SuperUser 와 데이터·플래그가 동일합니다. 자세히는 [관리자 콘솔 개요 · 초기(루트) SuperUser](20-admin-overview.md#permission-tiers).
- **"Agent 개발자(Agent Developer)" 는 등급이 아닌 역할/권한 묶음입니다.** Agent 개발자에게는 등급으로는 여전히 Standard User(`is_superuser: false`) 가 부여되며, 그 위에 `main.agentflow:*` / `main.tool:*` / `main.knowledge:*` 같은 권한이 묶인 *역할* 이 부여돼 Agent 작업실의 제작·도구·지식 메뉴가 열립니다. 흐름은 [Agent 작업실 개요 · 사용자 유형](../user/10-getting-started.md#user-types) 참고.

### 권한 계산이 동작하는 방식 (요약)

화면이 메뉴·버튼을 보여줄지 결정할 때 솔루션 내부는 **항상 같은 순서로 검사** 합니다.

1. **`is_superuser` 가 `true` 면 즉시 통과** — SuperUser 는 등급 자체로 모든 관리 영역 접근이 허용됩니다.
2. 아니면 사용자의 **권한 배열(`permissions`)** 을 살펴 요청된 ABAC 키와 매칭합니다. `*:*` 또는 `admin.user:*` 같은 와일드카드도 함께 매칭됩니다.
3. 매칭되는 권한이 하나도 없으면 해당 메뉴·버튼은 **자동으로 숨겨집니다** — 클릭 가능한 채로 두지 않습니다.

이 게이트는 *관리 설정 모드 접근*([관리자 콘솔 개요](20-admin-overview.md)), *AI 거버넌스 메뉴*([AI 거버넌스](29-governance-dashboard.md)), *AI 모델 관리* 등 도메인별 진입점에 동일하게 적용됩니다. 즉 동일 SuperUser 등급 안에서도 어떤 권한·역할이 묶여 있는지에 따라 *보이는 화면이 달라집니다.*

!!! info "권한 등급 변경은 어디서?"
    Standard User ↔ SuperUser 전환은 [사용자 관리](21-user-management.md) 의 사용자 편집 모달 *사용자 유형* 셀렉트(`Standard` ↔ `Superuser`)에서 수행합니다. **역할 부여**는 본 챕터에서, **개별 ABAC 권한 부여**는 사용자 편집 모달의 *직접 권한* 영역 또는 본 챕터의 *역할 → 권한 매핑* 에서 처리합니다.

### 사용자에게 권한이 닿는 4가지 갈래 { #grant-paths }

실무에서 SuperUser 가 사용자에게 권한을 *어떻게* 부여할지 결정할 때는 다음 4가지 경로 중 하나(또는 조합)를 선택합니다. *합성은 시스템이 자동* 으로 처리하므로, 관리자는 *어디서 부여할지* 만 고르면 됩니다.

| # | 부여 방법 | 어디서 부여하나 | 대표 사용 사례 |
|---|---|---|---|
| 1 | **등급(Tier) 통째 부여** | [사용자 관리](21-user-management.md) → 사용자 편집 → *사용자 유형* `Superuser` | 신규 시스템 관리자·거버넌스 담당자 |
| 2 | **역할 + 그 역할에 묶인 권한** | 본 챕터 → *역할* 탭 → 역할 생성 → *권한* 버튼 | "분석가" 역할로 `main.knowledge:read` 등 일관 권한 묶음 부여 |
| 3 | **역할 간 상속 (Supervision)** | 본 챕터 → *권한 구조* 탭 → 상위/하위 관계 추가 | "팀장 ← 팀원" 으로 *팀장에게 팀원 권한까지 자동 포함* |
| 4 | **개별 사용자에게 직접 부여** | [사용자 관리](21-user-management.md) → 사용자 편집 → *직접 권한* | 한 사용자에게만 일시적으로 권한 1개 Grant / Deny |

!!! note "시스템이 자동으로 합성하는 최종 권한"
    위 4 갈래 중 어느 것을 쓰든, 사용자가 실제로 가지는 최종 권한은 시스템이 다음 공식으로 합성합니다:

    **① 등급 우선 통과** — SuperUser 면 무조건 전부 → 그렇지 않으면 **② 본인 역할들의 권한 ∪ ③ Supervision 상속 권한 ∪ ④ Direct grant − ④ Direct deny**.

    즉 *역할 부여 + 일부 사용자만 권한 1개 추가/차단* 같은 조합이 가능하며, *deny 는 grant 보다 우선* 합니다.

## 역할 목록 { #role-list }

좌측 메뉴 **관리 설정 → 사용자 / 접근제어 → 역할/권한 관리** (view ID `admin-role-management`) 를 선택합니다.

![역할/권한 관리 — 등록된 역할 목록과 각 역할에 할당된 사용자 수](images/admin-roles.png)

### 화면 한눈에

페이지 상단은 **2개 탭** + **2개 액션 버튼** + **검색** 으로 구성됩니다.

| 영역 | 내용 |
|---|---|
| 탭 — *역할* | 역할 목록·생성·편집·권한 부여·사용자 할당 (본 절·다음 절들) |
| 탭 — *권한 구조* | 역할 간 상하 관계(Supervision) 설정 ([권한 구조](#supervision)) |
| 버튼 — *권한 조회* | 사용자 입력 후 *최종 합성된 권한* 을 조회 — 별도 권한 (`admin.permission:*`) 보유 SuperUser 만 노출 |
| 버튼 — *역할 생성* | 새 역할 등록 모달 열기 ([역할 생성](#role-create)) |
| 검색 | 역할명·표시명 즉시 필터 |

### 역할 행이 가지는 정보

| 컬럼 | 설명 |
|---|---|
| **역할명 (영문)** | 시스템 식별자 — kebab-case 권장 (예: `analyst`, `content-manager`) |
| **표시명** | 한글 또는 화면 노출용 이름 (예: "분석가", "콘텐츠 관리자") |
| **권한 수** | 이 역할에 묶인 권한 개수 |
| **설명** | 한 줄 설명 (없으면 `-`) |
| **작업** | 행별 4개 버튼 — **사용자** / **권한** / **편집** / **삭제** |

## 역할 생성 { #role-create }

역할 생성은 **두 단계** 입니다 — 먼저 역할명을 등록하고, 그 다음 역할 행의 *권한* 버튼으로 권한 체크리스트를 부여합니다.

### 1단계 — 역할 등록

1. 역할 목록 우상단 **역할 생성** 버튼 클릭 (매뉴얼 이전 버전이 안내한 *"+ 역할 추가"* 라벨은 현재 stg 에서 **"역할 생성"** 으로 표시됩니다).
2. 모달에서 다음 항목 입력:
    - **역할명 (영문)** — 예: `content-manager` (중복 불가)
    - **표시명** — 예: `콘텐츠 관리자`
    - **설명** — (선택)
3. 모든 필수 항목이 채워지면 우하단 **생성** 버튼이 활성화 — 클릭하면 역할이 권한 0개로 등록됩니다.

![역할 생성 모달 — 역할명(영문) / 표시명 / 설명 입력 필드와 취소·생성 버튼](images/admin-role-create.png)

### 2단계 — 권한 부여

생성된 역할에 권한을 부여하려면 역할 목록 테이블에서 해당 행의 **권한** 버튼을 클릭합니다. 권한 체크리스트가 카테고리별(admin / main scope)로 펼쳐지며 필요한 권한을 체크 후 저장합니다.

!!! note "권한 체크리스트 화면 캡처는 별도 갱신 예정"
    역할 목록 행 우측 **권한** 버튼 클릭 시 펼쳐지는 *권한 카테고리 체크리스트* 화면의 캡처는 다음 회차에 보강 예정입니다.

### 자주 쓰는 시나리오 { #scenarios }

**케이스 1 — 신규 직무에 맞는 역할 신설** ("컴플라이언스 담당" 예시)

1. *역할* 탭 → **역할 생성** 버튼.
2. 모달 입력: 역할명 `compliance-officer` / 표시명 `컴플라이언스 담당` / 설명 "AI 사용 위험 검토, 감사 로그 정기 조회".
3. 행 우측 **권한** 버튼 → `admin.governance:*` + `admin.audit:read` + `admin.user:read` 체크 → 저장.
4. 행 우측 **사용자** 버튼 → 컴플라이언스팀 구성원 다중 선택 → 일괄 할당.

**케이스 2 — 팀 단위 권한 묶음** ("EC 개발팀" 예시 — stg 에 실제 등록된 역할)

1. *역할* 탭에서 기존 *EC Development Team 1* 행의 **권한** 버튼 → 묶음 확인 (현재 33개).
2. 신규 팀원이 입사하면 같은 행의 **사용자** 버튼 → 추가만 하면 권한 자동 부여.
3. 팀에 새 책임이 추가되면 *권한* 버튼으로 해당 권한 체크 → 같은 팀 전체에 즉시 반영.

**케이스 3 — 일회성·임시 권한** (한 사용자만 일시적으로)

- 본 화면이 아닌 [사용자 관리](21-user-management.md) 의 사용자 편집 모달 → *직접 권한* 영역에서 Grant / Deny.
- 임시 권한을 *역할* 로 만들면 다른 사용자에게도 영향 — 1명에만 적용할 때는 *직접 권한* 이 안전합니다.

## 권한 구조 (상위/하위 권한) { #supervision }

복잡한 조직에서는 역할 간 상하 관계(Supervision) 를 정의해 *상위 역할이 하위 역할 권한까지 자동 상속* 받게 할 수 있습니다.

- **상위 권한 (Supervisor Role)**: 더 큰 권한 집합을 가진 역할
- **하위 권한 (Target Role)**: 더 작은 권한 집합을 가진 역할

예: "팀장" 역할이 "팀원" 역할을 상속하면, 팀장에게는 *팀원 권한 + 팀장 고유 권한* 이 모두 적용됩니다. 팀원 권한이 추가·변경되면 팀장도 자동으로 같은 변경을 받습니다.

### 작업 흐름

1. *권한 구조* 탭 진입.
2. **+ 관계 추가** (또는 *일괄 생성*) → 상위 역할 1개 + 하위 역할 N개 선택.
3. 설명(선택) 입력 후 **저장**.

상위 역할에 할당된 사용자는 *즉시* 하위 권한까지 보유한 상태가 됩니다 — 재로그인이나 새로고침 불필요 (다만 본인 사이드바 갱신은 다음 페이지 이동 시 반영).

!!! warning "순환 상속은 차단됩니다"
    `A ← B ← C` 가 있을 때 `C ← A` 같은 순환 등록은 시스템이 차단합니다. *상위 ← 하위* 단방향 그래프만 허용됩니다.

!!! note "Supervision vs. 권한 직접 부여"
    같은 결과를 얻는 두 방법:
    - 팀장 역할에 직접 *팀원 권한 + 팀장 권한* 을 모두 부여
    - 팀장 ← 팀원 Supervision 한 줄 + 팀장 역할에 *추가 권한만* 부여
    
    *팀원 권한이 자주 바뀌는 환경* 이라면 Supervision 이 유지보수에 유리합니다 (팀원 권한만 갱신하면 팀장도 자동 반영).

## 사용자에게 역할 할당 { #user-assignment }

**방법 A — 역할 화면에서 (다수 사용자에게 한 번에)**

1. 역할 상세 화면에서 **할당된 사용자** 섹션 펼치기
2. **+ 사용자 추가** 클릭 → 사용자 검색·선택
3. **저장**

**방법 B — 사용자 화면에서 (한 사용자에게 여러 역할)**

1. 사용자 편집 모달의 **역할** 섹션
2. 역할 선택 (다중 선택 가능)
3. **저장**

## 권한 회수

| 작업 | 절차 |
|---|---|
| 역할에서 사용자 빼기 | 역할 상세 → **할당된 사용자** → 해당 사용자 옆 **× 제거** |
| 역할 자체 삭제 | 역할 목록에서 **삭제** — 단, 이 역할에 할당된 사용자가 있으면 차단됨 |

!!! warning "역할 삭제 전 확인"
    삭제 전 **할당된 사용자 수**를 확인하세요. 0이 아니면 먼저 사용자에서 해당 역할을 회수해야 합니다.

## 운영 권장사항

**역할 명명 (생성 단계)**

- [ ] 영문 식별자(`name`)는 *kebab-case* (예: `content-manager`) — DB 키로 영구 보존되므로 대소문자 일관성이 중요합니다.
- [ ] 표시명(`display_name`)은 조직의 *직무·부서명* 과 일치시키면 사용자 관리·감사 시 직관적입니다.
- [ ] 설명은 한 줄로 — "무엇을 하는 역할인가" 가 인수인계·검색 시 결정적입니다.

**권한 부여 (역할 운영)**

- [ ] *최소 권한 원칙* — 시작은 좁게, 필요할 때 추가. 광범위한 권한(예: `*:*`, `admin.*:*`) 은 SuperUser 외에는 부여하지 않습니다.
- [ ] *임시 권한은 별도 역할로* — 변동성이 큰 임시 권한은 *임시 역할* 에 모아 회수 시 일괄 처리. 사용자 *직접 권한* 은 1명에만 적용할 때만 사용합니다.
- [ ] *Direct deny 는 예외용* — 역할로 받은 권한 중 *특정 사용자만* 차단해야 할 때만. 자주 쓰면 권한 흐름이 추적 불가합니다.

**정기 점검 (분기 1회 권장)**

- [ ] 할당 사용자 0명 역할 — 의도된 *컨테이너* 인지, 정리 누락인지 검토 후 삭제 또는 안내.
- [ ] 권한 0개 역할 — 같은 기준으로 검토.
- [ ] Supervision 그래프 — 불필요한 상속 관계 제거. 깊이 3 이상이면 *직접 권한 부여* 가 더 단순한지 재검토.
- [ ] 사용 안 하는 권한 키 — 백엔드에 더 이상 사용 안 하는 권한이 카탈로그에 남아 있다면 *기술 부록* 의 안내에 따라 운영팀에 보고합니다.

## 자주 발생하는 문제 { #troubleshooting }

| 증상 | 원인 / 점검 |
|---|---|
| "권한이 있는데 사이드바 메뉴가 안 보여요" | 사이드바 가시성은 *권한 prefix 매칭* 기준. 예: `admin.user:read` 만 있으면 *사용자 관리* 사이드바는 보이고 *AI 거버넌스* 사이드바는 안 보임. 권한 카탈로그 ↔ 사이드바 매핑은 *기능 단위* 로 묶여 있습니다. |
| "역할은 부여했는데 메뉴가 안 열려요" | 사용자가 *재로그인 또는 토큰 갱신* 필요. 또는 SuperUser 가 아닌데 *등급(Tier) 으로 게이팅* 되는 메뉴(예: 관리 설정 모드 전체) 일 수 있음 — 그 경우 [사용자 관리](21-user-management.md) 에서 *사용자 유형* 을 `Superuser` 로 변경. |
| "역할 삭제 시 'X명이 할당되어 있어 삭제할 수 없습니다'" | 의도된 안전장치. 역할 행의 **사용자** 버튼으로 진입해 사용자 0명으로 만든 뒤 다시 시도. |
| "한 사용자만 임시로 권한 끄고 싶어요" | [사용자 관리](21-user-management.md) → 편집 → *직접 권한* 영역에서 해당 권한을 **Deny** 로 설정. *역할 → grant* 와 *직접 → deny* 가 있으면 deny 가 우선 적용됩니다. |
| "새 기능 권한이 카탈로그에 안 보여요" | 권한 정의 자체는 *솔루션 빌드에 포함된 권한 카탈로그 파일* 한 곳에서 관리됩니다 (자세히는 [기술 부록](#technical-note)). 신규 기능 권한이 부여 UI 에 안 보이면 운영팀·플랫폼 담당에 문의하세요. |
| "권한 조회 버튼이 안 보여요" | *권한 조회* 는 별도 권한(`admin.permission:*`)이 부여된 SuperUser 에게만 노출됩니다. 본인이 SuperUser 인지, 이 권한이 본인 역할에 포함됐는지 확인. |

## 기술 부록 { #technical-note }

운영팀·플랫폼 담당자가 알면 좋은 내부 동작 — 일반 관리자는 본 절을 *읽고만* 두어도 됩니다.

- **권한 정의의 단일 진실원**: 솔루션 빌드 내부의 *권한 카탈로그 파일* 한 곳에서 모든 ABAC 키와 사이드바·탭 매핑이 관리됩니다. 백엔드 `@require_perm()` 데코레이터와 1:1 동기화되어야 하며, *새 권한 추가는 개발 사이드 작업* 입니다.
- **키 형식**: `{scope}.{domain}[-{subdomain}]:{action}` — 예 `admin.user:create`, `main.agentflow:manage`, `admin.governance-pii:read`.
- **와일드카드 매칭**: `*:*` (전체), `admin.user:*` (한 리소스의 모든 action). 매칭은 항상 *prefix startsWith* 기준이며, 사용자 권한 배열 중 *어느 하나라도* 해당 prefix 로 시작하면 통과합니다.
- **사용자 최종 권한 합성**: 백엔드가 단일 API (`GET /api/admin/permissions/resolve/{userId}`) 로 합성된 결과를 반환합니다. 클라이언트는 합성 로직을 다시 돌리지 않고, 받은 *역할 배열·권한 배열·`is_superuser`* 를 그대로 사용합니다.
- **숨김(폐기) 권한**: DB 에 잔존하는 폐기 권한은 *숨김 키 목록* 으로 UI 노출만 차단됩니다. 운영 영향 없이 안전하게 사라집니다.
