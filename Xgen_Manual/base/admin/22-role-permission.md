# 역할/권한 관리

본 챕터는 솔루션의 **권한 모델 전체 구조** 와, 조직 업무 단위에 맞춘 **역할(Role)** 정의·할당 절차를 다룹니다.

## 권한 모델 — 3 레이어로 분리되어 있다 { #permission-model }

XGEN 의 권한 체계는 *등급 / 역할 / 권한* 세 레이어가 **각각 독립적으로 동작**합니다. 흔히 한 단어로 묶여 쓰이지만 실제로는 의미·범위·관리 방법이 다릅니다.

```
┌──────────────────────────────────────────────────────────────────┐
│ 1. 권한 등급 (Tier)        ← 사용자가 "관리 설정" 모드에 들어갈  │
│    Standard User / SuperUser     수 있는가? 시스템이 정한 2단계  │
└──────────────────────────────────────────────────────────────────┘
                              ↓ SuperUser 위에서 의미를 갖는
┌──────────────────────────────────────────────────────────────────┐
│ 2. 역할 (Role)             ← 같은 SuperUser 라도 어떤 관리 메뉴 │
│    시스템 관리자 / 거버넌스       을 보는가? 조직이 자유롭게 정의 │
│    담당자 / 분석가 / 운영자 등                                    │
└──────────────────────────────────────────────────────────────────┘
                              ↓ Standard User 에게도 부여 가능한
┌──────────────────────────────────────────────────────────────────┐
│ 3. 권한 (Permission)       ← 각 화면·버튼·탭 단위 세밀한 잠금    │
│    admin.user:read,              ABAC 키. 역할에 묶이거나 사용자 │
│    main.agentflow:manage 등      에게 직접 부여                  │
└──────────────────────────────────────────────────────────────────┘
```

### 한눈 비교표

| 레이어 | 누가 정하나 | 값의 형태 | 예시 | 어디서 효과 |
|---|---|---|---|---|
| **권한 등급** | 시스템 (고정 2단계) | 단일 boolean 플래그 (`is_superuser`) | Standard User / SuperUser | 좌상단 **관리 설정** 모드 전환 버튼 가시성 |
| **역할** | 조직이 자유롭게 정의 | 다중 라벨 배열 | 시스템 관리자, 거버넌스 담당자, 분석가, 운영자, 컴플라이언스 담당, … | SuperUser 사이드바의 노출 메뉴 범위 |
| **권한** | 권한 카탈로그(고정) → 역할 또는 개별 사용자에 매핑 | ABAC 키 배열 | `admin.user:read`, `main.agentflow:manage`, `admin.governance-pii:*` | 화면 안의 섹션·탭·버튼 단위 게이팅 |

### 자주 헷갈리는 경계 정리

- **"관리자(Administrator)" 라는 중간 등급은 존재하지 않습니다.** 좌상단 *관리 설정* 모드에 들어갈 수 있는 사람은 모두 `is_superuser: true` 인 **SuperUser** 한 등급이며, 그 안에서 *시스템 관리자* / *거버넌스 담당자* 같은 **역할** 로 책임이 갈립니다. 등급 전체 정의는 [권한 등급 — 2단계 모델](20-admin-overview.md#permission-tiers), 두 역할의 메뉴 범위 차이는 [관리자 콘솔 개요 도입부](20-admin-overview.md) 의 *SuperUser 권한 필요* 안내를 참고하세요.
- **"초기(루트) SuperUser" 도 별도 등급이 아닙니다.** 솔루션 최초 설치 후 SuperUser 가 한 명도 없을 때 부트스트랩 경로(`/admin/create-superuser`) 로 만든 첫 SuperUser 일 뿐이며, 이후 사용자 관리 모달에서 *유형 = Superuser* 로 추가되는 SuperUser 와 데이터·플래그가 동일합니다. 자세히는 [관리자 콘솔 개요 · 초기(루트) SuperUser](20-admin-overview.md#permission-tiers).
- **"Agent 개발자(Agent Developer)" 는 등급이 아닌 역할/권한 묶음입니다.** Agent 개발자에게는 등급으로는 여전히 Standard User(`is_superuser: false`) 가 부여되며, 그 위에 `main.agentflow:*` / `main.tool:*` / `main.knowledge:*` 같은 권한이 묶인 *역할* 이 부여돼 Agent 작업실의 제작·도구·지식 메뉴가 열립니다. 흐름은 [Agent 작업실 개요 · 사용자 유형](../user/10-getting-started.md#user-types) 참고.

### 권한 계산이 동작하는 방식 (요약)

화면이 메뉴·버튼을 보여줄지 결정할 때 솔루션 내부는 **항상 같은 순서로 검사** 합니다.

1. **`is_superuser` 가 `true` 면 즉시 통과** — SuperUser 는 등급 자체로 모든 관리 영역 접근이 허용됩니다.
2. 아니면 사용자의 **권한 배열(`permissions`)** 을 살펴 요청된 ABAC 키와 매칭합니다. `*:*` 또는 `admin.user:*` 같은 와일드카드도 함께 매칭됩니다.
3. 매칭되는 권한이 하나도 없으면 해당 메뉴·버튼은 **자동으로 숨겨집니다** — 클릭 가능한 채로 두지 않습니다.

이 게이트는 *관리 설정 모드 접근*([관리자 콘솔 개요](20-admin-overview.md)), *AI 거버넌스 메뉴*([AI 거버넌스](29-governance-dashboard.md)), *AI 모델 관리* 등 도메인별 진입점에 동일하게 적용됩니다. 즉 동일 SuperUser 등급 안에서도 어떤 권한·역할이 묶여 있는지에 따라 *보이는 화면이 달라집니다.*

!!! info "권한 등급 변경은 어디서?"
    Standard User ↔ SuperUser 전환은 [사용자 관리](21-user-management.md) 의 사용자 편집 모달 *사용자 유형* 셀렉트(`Standard` ↔ `Superuser`)에서 수행합니다. **역할 부여**는 본 챕터에서, **개별 ABAC 권한 부여**는 사용자 편집 모달의 *직접 권한* 영역 또는 본 챕터의 *역할 → 권한 매핑* 에서 처리합니다.

## 역할 목록

좌측 메뉴 **관리 설정 → 사용자 / 접근제어 → 역할/권한 관리**를 선택합니다.

![역할/권한 관리 — 등록된 역할 목록과 각 역할에 할당된 사용자 수](images/admin-roles.png)

각 역할은 다음 정보를 가집니다.

| 항목 | 설명 |
|---|---|
| 역할명 | 영문 식별자 (예: `analyst`, `operator`) |
| 표시명 | 한글 이름 (예: "분석가", "운영자") |
| 권한 | 이 역할이 가지는 기능 접근 권한 목록 |
| 상위 권한 | (선택) 이 역할의 상위 역할 |
| 할당된 사용자 | 현재 이 역할을 가진 사용자 목록 |

## 역할 생성

1. 역할 목록 우상단 **+ 역할 추가** 버튼 클릭
2. 다음 항목 입력
    - 역할명 (영문, 중복 불가)
    - 표시명 (한글)
    - 설명 (선택)
3. **권한 선택** 영역에서 부여할 기능 권한 체크
4. **저장**

!!! note "역할 추가 화면 캡처는 별도 갱신 예정"
    "+ 역할 추가" 클릭 후 노출되는 권한 체크리스트 화면의 캡처는 다음 회차에 보강 예정입니다.

## 권한 구조 (상위/하위 권한)

복잡한 조직에서는 역할 간 상하 관계를 정의할 수 있습니다.

- **상위 권한 (Supervisor Role)**: 더 큰 권한 집합을 가진 역할
- **하위 권한 (Target Role)**: 더 작은 권한 집합을 가진 역할

예: "팀장" 역할은 "팀원" 역할의 모든 권한을 포함하고 추가로 "팀원 활동 모니터링" 권한을 가집니다. 상위 역할에 할당된 사용자는 하위 역할의 권한을 자동으로 상속받습니다.

## 사용자에게 역할 할당

**방법 A — 역할 화면에서 (다수 사용자에게 한 번에)**

1. 역할 상세 화면에서 **할당된 사용자** 섹션 펼치기
2. **+ 사용자 추가** 클릭 → 사용자 검색·선택
3. **저장**

**방법 B — 사용자 화면에서 (한 사용자에게 여러 역할)**

1. 사용자 편집 모달의 **역할** 섹션
2. 역할 선택 (다중 선택 가능)
3. **저장**

## 권한 회수

| 작업 | 절차 |
|---|---|
| 역할에서 사용자 빼기 | 역할 상세 → **할당된 사용자** → 해당 사용자 옆 **× 제거** |
| 역할 자체 삭제 | 역할 목록에서 **삭제** — 단, 이 역할에 할당된 사용자가 있으면 차단됨 |

!!! warning "역할 삭제 전 확인"
    삭제 전 **할당된 사용자 수**를 확인하세요. 0이 아니면 먼저 사용자에서 해당 역할을 회수해야 합니다.

## 운영 권장사항

- **역할명 일치** — 조직의 직무·부서 명칭과 역할명을 일치시키면 직관적입니다.
- **임시 권한은 별도 역할로** — 변동성이 큰 임시 권한은 사용자에 직접 부여하지 말고 별도 "임시" 역할에 모아 회수 시 일괄 처리합니다.
